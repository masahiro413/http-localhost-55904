# DESIGN.md — 設計ドキュメント

このファイルは、本アプリを **「なぜこの形に設計したか」** を記録するものである。

- [SPEC.md](SPEC.md)：**何を**作るか（仕様）
- [TASKS.md](TASKS.md)：**どう**作るか（手順）
- **DESIGN.md（このファイル）：なぜ**その設計にしたか（思想・構造・見た目の判断根拠）

学習が主目的のプロジェクトのため、「動く理由」だけでなく「そう決めた理由」を残す。

---

## 1. 設計思想（3つの原則）

1. **少ない機能を、きれいに作り切る。**
   機能を増やすより、追加・完了・削除という最小の機能を、読みやすいコードと
   心地よい UI で仕上げることに価値を置く。

2. **素の技術で、仕組みを見通せるようにする。**
   フレームワークやビルドツールを使わず、HTML / CSS / JavaScript を直に書く。
   「魔法」を減らし、DOM・イベント・localStorage の動きを自分の目で追えるようにする。

3. **状態は1つ、流れは一方向。**
   画面の元になるデータ（タスク配列）を唯一の情報源とし、
   「状態を変える → 保存する → 描き直す」という一方向の流れだけで動かす。
   これにより、画面と保存データが食い違うバグを構造的に防ぐ。

## 2. アーキテクチャ設計

### 2.1 ファイル構成（関心の分離）

| ファイル | 責務 |
|---|---|
| [index.html](index.html) | 画面の**構造**（何があるか） |
| [style.css](style.css) | 画面の**見た目**（どう見えるか） |
| [app.js](app.js) | 画面の**振る舞い**（どう動くか） |

3つに分けるのは、初心者が「構造・見た目・振る舞い」を別々の関心として
体で覚えるための、王道の学習構成でもある。

### 2.2 一方向データフロー（設計の心臓部）

```
   ユーザー操作
       │
       ▼
   tasks 配列を変更   ← 唯一の情報源（Single Source of Truth）
       │
       ▼
   save()             ← localStorage に保存
       │
       ▼
   render()           ← tasks から画面を組み立て直す
```

- **なぜこうするか**：DOM を直接ちまちま書き換えると、「配列は消したのに画面には残る」
  といった不整合が起きやすい。常に `tasks` を正として `render()` で描き直せば、
  画面は必ず配列と一致する。考えることが減り、バグが減る。
- **トレードオフ**：毎回リストを作り直すため、大量要素では非効率。
  だが本アプリの規模（個人利用・少数タスク）では、単純さの利点が上回る。
  実装は [app.js](app.js) の `render()` を参照。

### 2.3 状態の持ち方

- タスクは配列 `tasks`（`{ id, text, createdAt }` の並び）として持つ。
- **完了フラグは持たない。** 完了したタスクは配列から取り除くため、
  「配列にあるもの＝未完了のもの」という不変条件が常に成り立つ。
  状態がシンプルになり、フィルタ処理も不要になる。
- 完了の Undo 用に、`pending`（保留中の1件）という**一時状態**だけを別に持つ（§4.2）。

## 3. データ設計の意図

- **`id`**：各タスクを一意に識別する。`crypto.randomUUID()` を使い、
  無い環境では時刻ベースにフォールバック。DOM とデータを結び付ける鍵になる。
- **`text`**：`trim()` 済みの1行テキスト。属性はこれ以上増やさない（ミニマル維持）。
- **`createdAt`**：作成時刻。並び順の根拠・デバッグ用。表示はしない。
- **保存形式**：配列をそのまま `JSON.stringify` して localStorage の
  `simple-todo.tasks` キーに入れる。読み込み時は壊れていても空配列に倒し、
  **アプリが必ず起動できる**ことを最優先にする（防御的読み込み）。

詳細な永続化仕様は [SPEC.md](SPEC.md) §5 を参照。

## 4. インタラクション設計

### 4.1 追加

- 入力欄の **Enter** と「追加」ボタンの両方で追加できる（`<form>` の submit に集約）。
- 空文字（trim 後）は追加しない。追加後は入力欄を空にしフォーカスを戻し、
  **連続入力のリズム**を止めない。
- 新規は**末尾に追加**。上から片付ける「待ち行列」のメンタルモデルに合わせる。

### 4.2 完了 ＝「即消え」＋「数秒の取り消し」（設計の要）

完了は、2つの相反する要求を同時に満たす必要がある。

- 達成したものは**すぐ視界から消したい**（潔さ）。
- でも**押し間違いは救いたい**（安全網）。

そこで「**保留（pending）→ 確定 or 取り消し**」というパターンを採用した。

```
完了クリック
   │  tasks から即除去 → save → render（画面から消える）
   ▼
pending に退避（task・元の位置・タイマーを保持）＋ Undo トースト表示
   │
   ├─ 「元に戻す」を押す → 元の位置へ挿し戻して描き直す（復活）
   ├─ 5秒たつ           → finalizePending（確定・もう戻せない）
   └─ 次の完了が来る     → 直前の保留を確定してから、新しい完了へ
```

- **設計判断**：確認ダイアログ（「完了しますか？」）は毎回の一手間が煩わしいので採らない。
  代わりに「まず消す・数秒だけ戻せる」で、潔さと安全網を両立させた。
- 関連コード：[app.js](app.js) の `completeTask` / `finalizePending` / `undoComplete`。

### 4.3 削除 ＝「慎重な操作」

- 削除は「そもそも不要／間違い」を取り除く操作で、完了とは**意味が異なる**。
- こちらは Undo ではなく**確認ダイアログ**を挟む（`window.confirm`）。
  破壊的だからこそ、実行前に一度立ち止まらせる。
- 「完了＝気軽・即実行」「削除＝慎重・確認」という**操作の重さの差**を、
  そのまま UX に反映している。

## 5. ビジュアル設計

方向性は「**カード型・親しみやすい／毎日開きたくなる見た目**」。

### 5.1 カラーシステム（CSS 変数）

- 色は個別に指定せず、[style.css](style.css) 冒頭の**カスタムプロパティ**に集約する
  （`--bg` `--surface` `--text` `--muted` `--accent` `--border` など）。
- **利点**：色を変えたいときは1か所を直すだけで全体に反映される。
  また、テーマ切り替えが「変数の値の差し替え」だけで済む。

### 5.2 ライト／ダークテーマ

- OS の設定に**自動追従**する（`@media (prefers-color-scheme: dark)` で変数を上書き）。
- **なぜ自動追従か**：毎日開くアプリなので、時間帯や環境に合わせて
  目に優しい方が自然に選ばれるべき。手動トグルは発展課題（§7）に回した。

### 5.3 見た目のトーン

- **カード**：角丸（`border-radius`）＋柔らかい影（`box-shadow`）で、
  緊張感のない親しみやすさを出す。
- **アクセント色**：温かみのあるコーラル系。事務的すぎず、開くのが少し楽しくなる色。
- **余白・行間**：ゆったり取り、詰め込まない。1画面の情報量より心地よさを優先。
- **さりげない動き**：カード出現や トースト表示に短いアニメーションを添え、
  変化を目で追えるようにする（過剰にはしない）。

## 6. アクセシビリティ設計

- 操作要素はすべて `<button>` にし、**クリックにもキーボードにも**対応する。
- アイコンだけのボタン（完了・削除）には `aria-label` を付け、
  スクリーンリーダーでも意味が分かるようにする。
- 件数表示とトーストには `aria-live` を付け、状態変化を読み上げ可能にする。
- キーボード操作時は `:focus-visible` でフォーカス位置をはっきり示す。

## 7. 検討した代替案と、採らなかった理由

設計は「選ばなかった道」を残すと後から理解しやすい。

| テーマ | 採用 | 見送った案とその理由 |
|---|---|---|
| 技術 | 素の HTML/CSS/JS | React 等：学習目的では「素で理解する」方を優先。依存も増える |
| 保存先 | localStorage | サーバー同期：シンプルさと明確にトレードオフ。個人・単一端末なら不要 |
| 完了の誤操作対策 | 数秒の Undo | 確認ダイアログ：毎回の一手間が煩わしく、潔さを損なう |
| 削除の安全策 | 確認ダイアログ | Undo：削除は意味的に「間違いの除去」。先に止める方が自然 |
| 完了の管理 | 配列から除去 | 完了フラグ＋非表示：状態が増え、フィルタも必要になる |
| テーマ | OS 自動追従 | 手動トグル：まずは自動で十分。トグルは発展課題へ |
| 編集 | コア外（発展課題） | 最初から実装：ミニマル維持。学習の伸びしろとして後回し |

これらのトレードオフの背景は [SPEC.md](SPEC.md) §8 にも記録している。

## 8. 将来この設計が壊れそうな点（拡張時の注意）

- **描き直し方式**：タスクが非常に多くなると全再描画が重くなる。
  そのときは差分更新やフレームワーク導入を検討する（現状は不要）。
- **単一の tasks 配列**：属性（締切・優先度など）が増えると、
  データモデルと描画の両方が膨らむ。増やすなら [SPEC.md](SPEC.md) の非スコープを見直す。
- **localStorage 依存**：同期や大容量が必要になった時点で、
  保存層（save/load）を差し替えられるよう、その2関数に処理を閉じ込めてある。
