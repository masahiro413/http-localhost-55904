# SPEC.md — 個人用シンプル ToDo アプリ 仕様書

## 1. 目的と背景

PC の Web ブラウザ上で動く、個人用のシンプルなタスク管理アプリを作る。
機能は「タスクの**追加・完了・削除**」に絞る。

ただし、この草案の裏にある本当の目的は **「Web の基礎を素の状態で学ぶこと」** である。
アプリ自体は意図的にミニマルに保ち、以下を学習の対象とする。

- HTML / CSS / JavaScript を、フレームワークなしの「素の状態」で書く
- DOM 操作・イベント処理・`localStorage` を直に理解する
- コードの読みやすさと、**毎日開きたくなる見た目（CSS）** を両立させる

したがって本アプリは「機能を増やすこと」ではなく、
**少ない機能を、きれいなコードと心地よい UI で作り切ること** に価値を置く。

---

## 2. スコープ

### 2.1 対象（やること）

- タスクの追加（1行のテキスト）
- タスクの完了（一覧から即座に消える／数秒間は取り消し可能）
- タスクの削除（完了とは別の操作。実行前に確認を挟む）
- `localStorage` によるデータの永続化（同一ブラウザ内）
- カード型の親しみやすい UI／OS 設定に追従するライト・ダークテーマ

### 2.2 対象外（やらないこと）

以下は「シンプルさ」を守るため、意図的にスコープ外とする（将来の拡張余地は §9 に記録）。

- サーバー／アカウント／複数デバイス間の同期
- 締め切り・優先度・カテゴリ・メモなど、テキスト以外の属性
- 完了タスクの履歴・集計・振り返り
- 検索・フィルタ・タグ
- タスクの編集（※「発展課題（任意）」として §6 の Step で扱う）

---

## 3. 用語

| 用語 | 定義 |
|---|---|
| タスク | ユーザーがこなすべき事柄。本アプリでは「1行のテキスト」で表す |
| 未完了リスト | 画面に表示される、まだ完了していないタスクの一覧 |
| 完了 | タスクをやり遂げて、一覧から取り除く操作 |
| Undo（元に戻す） | 完了直後の数秒間だけ、完了を取り消してタスクを一覧に戻せる救済機能 |
| 削除 | タスクを一覧から取り除く操作。完了とは意味が異なる（§4.4 参照） |

---

## 4. 機能要件

### 4.1 タスクの追加

- 画面上部（または下部）の**入力欄**にテキストを打ち、**「追加」ボタン**または **Enter キー**で追加する。
- 追加されたタスクは、未完了リストの**一番下に積まれる**（古いものが上、新しいものが下）。
  - 理由：上から順に片付けていく「待ち行列（キュー）」の感覚に合うため。
- **バリデーション**
  - 前後の空白を取り除いた結果が空文字の場合は、追加しない（入力欄は保持）。
  - 文字数の上限は設けない（長い場合は折り返して表示する）。
- 追加後は入力欄を空にし、フォーカスを入力欄に戻す（連続入力しやすくする）。

### 4.2 タスクの完了

- 各タスクの**チェックボックス（または完了ボタン）**をクリックすると、そのタスクは
  **未完了リストから即座に消える**。
- 完了したタスクの履歴は残さない（振り返り機能は持たない）。
- ただし、**押し間違いの安全網**として、完了直後に画面下部へ
  **「元に戻す（Undo）」トースト**を数秒間（既定：**5秒**）表示する。
  - トースト内の「元に戻す」を押すと、そのタスクは元の位置に近い形で未完了リストへ戻る。
  - トーストが時間切れになる、または次の完了操作が行われると、
    そのタスクは `localStorage` からも**完全に削除**される（＝復元不可になる）。

### 4.3 タスクの削除

- 各タスクには、完了とは**別の「削除」ボタン**を用意する。
- 削除は「そもそも不要だった／間違って追加した」タスクを取り除くための操作であり、
  完了より**慎重に**扱う。
- 削除ボタンを押すと、**確認**（例：`window.confirm` による確認ダイアログ、
  もしくは「本当に削除しますか？」の確認 UI）を挟み、承認された場合のみ削除する。
- 削除には Undo を用意しない（確認済みのため）。

### 4.4 「完了」と「削除」の違い（設計意図）

両者はどちらも「タスクを一覧から消す」が、**意味が異なる**ため挙動を分ける。

| | 完了 | 削除 |
|---|---|---|
| 意味 | やり遂げた（達成） | そもそも不要／間違い |
| 操作の重さ | 気軽・即実行 | 慎重・確認を挟む |
| 取り消し | 数秒間の Undo あり | なし（確認済み） |

### 4.5 （発展・任意）タスクの編集

- コア機能には含めないが、学習の伸びしろとして**インライン編集**を任意で実装できる。
- 例：タスク文字列を**ダブルクリック**すると、その場で編集可能な入力欄に変わり、
  Enter または フォーカス外れで確定する。空文字確定は無視する。

## 5. データ設計

### 5.1 データモデル

1つのタスクは次の形の JavaScript オブジェクトで表す。

```js
{
  id: string,        // 一意な識別子（例: crypto.randomUUID() または Date.now() ベース）
  text: string,      // タスク本文（trim 済み・1行）
  createdAt: number  // 作成時刻（Date.now() のミリ秒。並び順・デバッグ用）
}
```

- タスク全体は、この配列 `Task[]` として保持する。
- **完了フラグは持たない**。完了したタスクは配列から取り除かれるため、
  「未完了のものだけが常に配列に存在する」状態を保つ。

### 5.2 永続化仕様（localStorage）

- 保存先：ブラウザの `localStorage`。
- キー名：`"simple-todo.tasks"`（他アプリと衝突しない名前空間付き）。
- 保存形式：`JSON.stringify(tasks)` で文字列化して保存。
- 読み込み：起動時に `JSON.parse` で復元する。
  - 値が無い／壊れている場合は、空配列 `[]` として扱い、アプリが必ず起動できるようにする。
- 保存タイミング：タスク配列を変更する操作（追加／完了確定／Undo／削除）のたびに保存する。

## 6. UI / UX 仕様

### 6.1 全体の見た目

- 方向性：**カード型・親しみやすい**デザイン。
  - 各タスクは角丸のカードとして表示し、柔らかい影（`box-shadow`）を添える。
  - 余白・行間をゆったり取り、毎日開いても疲れない、心地よい見た目にする。
- レイアウト：中央寄せ・最大幅を設けた1カラム（PC の Web ブラウザを主対象）。

### 6.2 テーマ（ライト／ダーク）

- OS のテーマ設定に**自動追従**する（CSS の `prefers-color-scheme` を使用）。
  - ライトモード：明るい背景＋落ち着いた文字色。
  - ダークモード：暗い背景＋目にやさしいコントラスト。
- 色は CSS カスタムプロパティ（変数）で定義し、テーマごとに値を切り替える。

### 6.3 画面の構成要素

1. **ヘッダー**：アプリ名（例：`My Tasks`）と、必要なら未完了件数の表示。
2. **入力エリア**：テキスト入力欄＋「追加」ボタン。
3. **タスクリスト**：カードが縦に並ぶ。各カードは
   「チェックボックス／完了」「タスク本文」「削除ボタン」で構成。
4. **空状態（Empty State）**：タスクが1件も無いとき、
   さりげないメッセージ（例：「タスクはありません。今日やることを追加しましょう」）を表示する。
5. **Undo トースト**：完了直後に画面下部へ現れ、数秒後に消える一時的な通知。

### 6.4 操作方法（キーボード対応）

- 入力欄で **Enter** → タスク追加。
- 各操作要素（チェックボックス・削除・Undo）は**ボタン要素**とし、
  クリックだけでなくキーボードでも操作できるようにする。
- 適切なラベル（`aria-label` 等）を付け、最低限のアクセシビリティを確保する。

## 7. 対象環境

- PC の**モダンブラウザ**：Chrome / Edge / Firefox / Safari の各最新版。
- ビルドツール不要。`index.html` をブラウザで直接開くだけで動作する
  （開発時は VS Code の Live Server 等を使ってもよい）。
- モバイル対応は必須要件ではない（PC 利用を前提とする）。

## 8. 懸念点とトレードオフ（明示的な記録）

「シンプルさ」を優先した結果として受け入れる制約を、あえて明文化しておく。

- **データ消失リスク**：`localStorage` に保存するため、ブラウザのキャッシュ／サイトデータを
  削除するとタスクは消える。バックアップ機能は持たない。
- **単一デバイス・単一ブラウザ限定**：別の PC・別のブラウザからは同じタスクを見られない。
  複数デバイス同期はサーバーが必要になり、シンプルさと明確にトレードオフの関係にある。
- **履歴を残さない**：完了タスクは復元期限を過ぎると消えるため、
  「これまで何を終わらせたか」を振り返ることはできない。
- **Undo の実装コスト vs 安全網**：完了で即消える潔さを保ちつつ誤操作を救うため、
  一時状態とタイマー（`setTimeout`）を扱う。単純な即時削除よりわずかに複雑になるが、
  日常利用の安心感を優先して採用する。
- **フレームワーク不使用のトレードオフ**：学習しやすく寿命が長い（依存が腐らない）反面、
  将来 UI が複雑化した場合の状態管理は自前になる。本アプリの規模では利点が上回ると判断。

## 9. 将来の拡張余地（スコープ外・参考）

現時点では実装しないが、必要になったときの拡張候補として記録しておく。

- 複数デバイス間の同期（サーバー or クラウドストレージ）
- 締め切り・優先度・カテゴリなどの属性追加
- タスクの編集（インライン編集。§4.5 参照）
- 手動での並び替え（ドラッグ＆ドロップ）
- 完了タスクの履歴・集計・振り返り
- データのエクスポート／インポート（バックアップ手段）
